مشکلات دنیای واقعی

تنظیمات جزو لاینفک هر برنامه است. و هر برنامه پدرمادردار، مسلما تنظیمات زیادی هم داره. نصب یه برنامه نباید حجم زیادی از این تنظیمات رو از کاربر بپرسه. در واقع بهتره که کاربر بعد از نصب برنامه با تنظیمات پیش فرض، دسترسی ساده و سریع داشته باشه به تنظیمات تا اگه خواست چیزی رو طبق میل خودش عوض کنه، به راحتی این کار رو انجام بده. گیت که تقریبا مشهورترین VCS محسوب میشه، در نصبش 9 تا تنظیم رو میپرسه. این به نظرم یه مشکل واقعی محسوب میشه.

رو یوتوب ببین

مشکل در طراحی مناسب UI/UX حتی در سطوح تراز اول دنیا هم دیده میشه. شرکت Adobe چند دهه است که فعاله و طبیعتا انتظار میره که یه همچین موضوع ساده ای رو حل کرده باشه.

یه فتوشاپ باز کنین. یه متن ساده توش بنویسین، اما از کادر متن خارج نشین. حالا برین دکمه close رو بزنین. هم در سطح اپلیکیشن، هم در سطح document. هیچ کدوم هیچ واکنشی نشون نمیدن. این از افتضاح یه سر اونورتره. اینکه این برنامه در چه stateی قرار داره از نگاه من که کاربر عادی فتوشاپ هستم اصلا مهم نیست. اینکه من الان میخوام برنامه رو ببندم، روی دکمه close کلیک میکنم، ripple رو میبینم و متوجه میشم که دکمه کلیک شده اما هیچ واکنشی از برنامه نمیبینم، این یعنی UX نامطلوب.

برنامه در هر حالتی که باشه، و هر logicی که تعبیه شده باشه برای closing، فشردن دکمه close صریح ترین حالت ممکنه که اراده کاربر تحقق پیدا کنه. کمترین کاری که فتوشاپ میتونه انجام بده اینه که اینجا یه پیام بده که مشکل چیه، و یه سری گزینه رو در اختیار کاربر بذاره.

ASP.NET Core Controllers

چونکه اول MVC و Web API از هم جدا بودن. MVC یه کلاس داشت به اسم Controller و Web API یه کلاس داشت به اسم ApiController. بعدا که یکی شدن فقط یه کنترلر وجود داشت به اسم Controller. اما این Controller الکی سنگین بود برای API و موضوعاتی تو دلش بود که به API ربطی نداشت، مثل ViewData. واسه همین بعدا یه Controller ساخته شد به اسم ControllerBase که فقط برای API باشه و سبک باشه. اما واقعا نامگذاری counter-intuitive شد. یعنی حتی در سطح کلان، شرکتی به بزرگی مایکروسافت گاهی توی change گیر میکنه و مجبور میشه کثیفی رو به جون بخره (بدهی فنی). نامگذاری intuitive میتونه اینطوری باشه:

  • Controller => RazorController
  • ControllerBase => Controller
