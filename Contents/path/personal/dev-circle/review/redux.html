Redux

  • Aweful, terribly misleading terminology
    • Action, Reducer are very very bad terms

      • Instead of action, I would use the term "event".

        For example in a todo app, instead of defining an action that is called "addTodo", I'll define an event called "addedTodo". Because based on passing parameters to actions and from actions to reducers, we're dealing with an actual event that has happened from user interaction or another source, rather than just describing an abstract idea.

        Actions describe what are about to happen. Events describe what happened.

        From redux docs:

        An action is a plain JavaScript object (notice how we donâ€™t introduce any magic?) that describes what happened.

      • Instead of reducer, I would use the term "handler". What reducer does, is to handle different actions in one central place. It's more like an orchestrator of handlers. When this event happened, change state based on these parameters. When that event happend, do so with those parameters, etc.
    • Action thunk is bad. I would use "event template".
    • Store is not that bad, but since people think of a shop when we say store, and also developers use other terms for this, I would prefer to use "container" or "repository" instead of store.
    • State, Dispatch and Connect are good terms
